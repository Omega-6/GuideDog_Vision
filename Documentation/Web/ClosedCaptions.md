# Closed Captions

## Overview

The closed captions feature transcribes speech, displaying it in a caption box at the bottom of the screen in real time. It uses the browser's built-in `SpeechRecognition` API in continuous mode with interim results enabled. No audio is sent to outside servers meaning transcription is handled entirely by the browser vendor's speech recognition service.

---

## Why Not Third-Party Automatic Speech Recognition (ASR)?

The `SpeechRecognition` API is the only audio transcription service available in a browser without streaming audio to an external service. Third-party ASR APIs like Whisper and Deepgram require sending audio over the network, causing latency, cost, and privacy concerns. The Web Speech API runs the transcription inside the browser with no code changes required, negating these issues.

The trade-off here is API availability because Firefox doesnâ€™t support `SpeechRecognition`. On iOS Safari, the API works, but it requires Siri to be enabled in device settings. These limitations are handled with spoken error messages regarding caption availability.

## Initialization (`initCaptionRecognition`)

Called the first time captions are enabled (lazily, not at startup). Creates a `SpeechRecognition` instance with:

r.continuous \= true;      // Don't stop after one phrase

r.interimResults \= true;  // Fire events for partial results too

r.lang \= 'en-US';

r.maxAlternatives \= 1;

### 

### Event Handlers

**`onresult`**

Iterates `e.results` from `e.resultIndex` onward, and appends final results to `state.captionFinalText`. This is sliced to the last 100 characters to prevent unbounded growth. Interim results, preliminary word guesses generated by the AI, are passed separately, and both are forwarded to `showCaption()`.

The 100-character limit is chosen against the caption box height, which is 16px font size with 1.4 line-height; 3 lines hold roughly 110â€“130 characters of typical spoken text. Slicing to 100 allows older speech to come out of the top of the box naturally, creating a smoother user experience.

**`onerror`**

Handles several distinct error cases:

- `no-speech` â€” benign timeout, `onend` will restart  
- `aborted` â€” the app called `stop()`, ignore  
- `network` â€” transient failure, warn and let `onend` restart  
- `not-allowed` â€” microphone permission denied; disable captions, update button state, speak an error  
- `service-not-allowed` â€” Safari with Siri disabled; disable captions, speak instructions

**`onend`**

If `state.captionEnabled` is still true, it schedules a restart after 300ms:

setTimeout(() \=\> { try { r.start(); } catch (\_) {} }, 300);

The 300ms gap gives the browser time to release the microphone before reopening it. Without the delay, `start()` throws on some browsers. The auto-restart means captions stay on indefinitely without manual intervention, even across network issues or brief silences.

## Caption Display (`showCaption`)

The `#captionBox` element is a fixed-width strip at the bottom of the screen, above the sound banner:

- Width: `min(78vw, 340px)`  
- Height: exactly `calc(3 Ă— 1.4em + 20px)` â€” three lines plus padding  
- `overflow: hidden` clips text that exceeds three lines  
- `display: flex; flex-direction: column; justify-content: flex-end` â€” newest text pins to the bottom edge

Content is placed inside a `.cap-inner` div. Final text is wrapped in `<span class="final">` (white). Interim text is wrapped in `<span class="interim">` (50% opacity, italic). When the inner div grows taller than the outer box, the top of the inner is clipped â€” exactly the oldest text â€” while the newest text remains visible at the bottom.

HTML is built via `escapeHTML()` which replaces `&`, `<`, and `>` to prevent injection through transcribed content.

### Auto-Hide

A hide timer is set on every `showCaption` call:

clearTimeout(state.captionHideTimer);

state.captionHideTimer \= setTimeout(() \=\> {

    state.captionFinalText \= '';

    hideCaption();

}, CONFIG.CAPTION\_HIDE\_DELAY); // 5000ms

If no new speech arrives within 5 seconds, the caption box fades out and `captionFinalText` is cleared. The next speech starts fresh.

## Toggle (`toggleCaptions`)

Called by:

- The CC chip in the GuideDog mode status bar (if present)  
- The CC button in the Assist mode bar  
- The voice command handler (keywords: "caption", "subtitle", "closed caption")

On enable:

1. Initializes `captionRecognition` if not yet created  
2. Sets `state.captionEnabled = true`  
3. Clears `state.captionFinalText`  
4. Calls `r.start()` (inside try-catch)  
5. Speaks "Live captions on." (suppressed in Assist mode since deaf users won't hear it â€” but this is still called; the `doSpeak` guard exits early for Assist mode)

On disable:

1. Calls `r.stop()` inside try-catch  
2. Calls `hideCaption()`  
3. Speaks "Live captions off."

## Caption Overlay Position

The `#captionOverlay` element is `position: fixed` at the bottom of the viewport, above all other content including the help and privacy dialogs (`z-index: 200`). It contains two elements stacked vertically:

1. `#soundBanner` â€” the sound detection event pill (blue background)  
2. `#captionBox` â€” the speech transcript strip (dark translucent background)

Both use `pointer-events: none`, so they never intercept taps meant for the UI beneath.

`bottom: max(20px, env(safe-area-inset-bottom))` keeps both elements above the home indicator on iPhone.

## Assist Mode Auto-Start

When `startAssistMode()` is called, captions start automatically after 600ms:

setTimeout(() \=\> {

    if (\!state.captionRecognition) initCaptionRecognition();

    state.captionEnabled \= true;

    const btn \= document.getElementById('btnAssistCC');

    btn.classList.add('on');

    btn.setAttribute('aria-pressed', 'true');

    try { state.captionRecognition?.start(); } catch (e) {}

    document.getElementById('assistStatusText').textContent \= 'Captions on';

    document.getElementById('assistStatusSub').textContent \= 'Speak and text will appear below';

}, 600);

The 600ms delay allows the mode transition animation time to finish and prevents a race condition where `start()` is called before the screen is fully rendered.

## Browser Compatibility Notes

| Browser | Caption support | Notes |
| :---- | :---- | :---- |
| Chrome (Android/Desktop) | Full | Works reliably |
| Safari (iOS/macOS) | With Siri enabled | `service-not-allowed` if Siri is off |
| Edge | Full | Uses same engine as Chrome |
| Firefox | Not supported | API absent; silently disabled |
| Samsung Internet | Partial | May require speech permission grant |

When the API is unavailable, `initCaptionRecognition()` returns `false`, a "Captions not supported on this device" message is spoken, and the CC button remains in its off state.

## Conflict with Voice Command Recognition

The voice command recognition instance (`state.recognition`) and the caption recognition instance (`state.captionRecognition`) are separate `SpeechRecognition` objects. They share the microphone at the OS level.

When the user taps the microphone button to issue a voice command in GuideDog mode, `startListening()` checks `state.isListening` and calls `stopListening()` if needed. If captions are enabled, they continue running in the background. On most browsers, two simultaneous `SpeechRecognition` instances share the audio stream without conflict. However, on some mobile browsers the second `start()` call may return a `not-allowed` error if the OS considers the mic already in use. This is handled by the `onerror` handler's restart logic.
